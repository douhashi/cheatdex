# ビジネスモデル / 利用者像

## 想定利用者

エミュレータでレトロ／旧世代ゲームをプレイし、チートコードを使うユーザー。

- 複数プラットフォーム（PS2 / PS / SNES など）のエミュレータを併用する
- 自分で集めたチートコードを整理・再利用したい
- エミュレータごとに手作業でチートファイルを編集するのが面倒

## 提供形態

- Web アプリケーション（ブラウザからアクセス）
- Google アカウントでログインし、自分のチートデータを管理する個人向けツール

## 主要ユースケース

1. **登録**: プラットフォーム → ゲーム → チートコードの順に登録する
2. **整理**: ゲーム単位でチートを束ね、使うものだけ ON にする
3. **書き出し**: 全体またはプラットフォーム単位で zip ダウンロードし、エミュレータに取り込む

## 課金 / 提供範囲

- 現時点では未定。まずは個人利用を前提とした機能提供から始める。

## 要確認の論点

### ON/OFF の粒度とデータ所有モデル

「各チートコードは ON/OFF が可能」の解釈で、データモデルと提供形態が変わる。

- **案A（採用提案）**: 各ユーザーが自分のチートコレクションを所有する。チートコードは登録したユーザーの所有物で、ON/OFF はそのレコードの属性。Google ログインのみで個人管理を行う想定と最も整合する。
- **案B**: チートコードを共有マスタとして持ち、ユーザーごとに有効/無効を中間テーブルで管理する。ユーザー間共有・公開を将来見据える場合に向く。

→ 「含まない」スコープに共有を置いているため、まず **案A** を採用する。共有要件が顕在化した時点で案B へ拡張する。

詳細なデータ構造は [../development/data-model.md](../development/data-model.md) を参照。
